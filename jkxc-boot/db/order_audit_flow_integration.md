# 订单审核流程与工商任务流程集成说明

## 概述

本文档说明订单审核流程与工商任务流程的集成方式，以及如何灵活配置审核流程。

## 流程说明

### 1. 订单创建流程

当创建订单时，系统会自动执行以下操作：

1. **创建订单主记录**：保存订单基本信息
2. **创建订单流程步骤**：自动创建"订单创建"步骤
3. **初始化审核流程**（如果订单配置了审核流程）：
   - 根据订单的 `auditProcessId` 查找审核流程配置
   - 为每个审核步骤创建审核记录（`gh_order_audit` 表）
   - 所有审核记录初始状态为 `pending`（待审核）
4. **创建工商任务**（如果交单方式是工商部）：
   - 自动创建工商任务记录（`gh_business_task` 表）
   - 任务初始状态为 `pending_manager_audit`（待工商经理审核）

### 2. 订单审核流程

订单审核流程与审核流程管理（`sys_audit_process`）配合工作：

1. **审核流程配置**：
   - 在"审核流程管理"中配置审核流程
   - 每个流程包含多个审核步骤（`sys_audit_step`）
   - 每个步骤关联一个审核角色（`sys_role`）
   - 支持并行审核（相同 `stepOrder` 的多个步骤）

2. **执行审核**：
   - 调用 `/order/audit` 接口执行审核
   - 参数：`orderId`、`roleId`、`auditStatus`（approved/rejected）、`auditRemark`
   - 系统会更新对应的审核记录
   - 检查当前步骤的所有角色是否都已审核通过
   - 如果所有步骤都通过，订单审核状态更新为 `approved`

3. **审核完成后的处理**：
   - 如果审核通过且交单方式是工商部，系统会创建或更新工商任务
   - 如果审核驳回，订单状态更新为驳回状态

### 3. 工商任务流程

工商任务流程包含以下状态和操作：

1. **工商经理审核**（`pending_manager_audit`）：
   - 工商经理可以审核任务
   - 审核通过后可以选择：
     - 放入公海（`public_sea`）：任何工商人员都可以接收
     - 指定工商人员（`assigned_to_me`）：指定特定工商人员处理

2. **公海待接收**（`public_sea`）：
   - 工商人员可以接收任务
   - 接收后任务状态变为 `task`

3. **待本人接收**（`assigned_to_me`）：
   - 被指定的工商人员可以接收任务
   - 接收后任务状态变为 `task`

4. **任务**（`task`）：
   - 工商人员填写成本类目和成本金额
   - 成本信息会自动同步到订单支出（`gh_order_expense` 表）
   - 可以执行以下操作：
     - 去交接：任务状态变为 `handover`
     - 转分配工商人员：重新分配给其他工商人员
     - 转为异常订单：选择放入问题任务或回收站

5. **交接**（`handover`）：
   - 工商人员确认交接完成
   - 任务状态变为 `completed`（已完单）

6. **已完单**（`completed`）：
   - 任务已完成，可以查看历史记录

7. **问题任务**（`problem_task`）：
   - 异常任务，需要特殊处理

8. **回收站**（`recycle_bin`）：
   - 已删除的任务，可以恢复

## 数据库表结构

### 1. 订单审核记录表（`gh_order_audit`）

记录订单审核流程的每一步审核：

- `order_id`：订单ID
- `audit_process_id`：审核流程ID
- `step_order`：审核步骤顺序
- `role_id`：审核角色ID
- `audit_status`：审核状态（pending/approved/rejected）
- `audit_user_id`：审核人ID
- `audit_time`：审核时间
- `audit_remark`：审核备注

### 2. 订单表（`gh_order`）

新增字段：

- `audit_process_id`：审核流程ID
- `current_audit_step`：当前审核步骤顺序
- `audit_status`：审核状态（pending/approved/rejected）

### 3. 工商任务表（`gh_business_task`）

记录工商任务的详细信息：

- `order_id`：关联订单ID
- `task_status`：任务状态
- `manager_audit_status`：经理审核状态
- `cost_category`：成本类目
- `cost_amount`：成本金额
- 其他相关字段...

## API 接口说明

### 1. 订单审核接口

**接口地址**：`POST /order/audit`

**请求参数**：
```json
{
  "orderId": "订单ID",
  "roleId": "审核角色ID",
  "auditStatus": "approved|rejected",
  "auditRemark": "审核备注"
}
```

**响应**：
```json
{
  "success": true,
  "message": "审核成功"
}
```

### 2. 获取订单审核记录

**接口地址**：`GET /order/getOrderAudits`

**请求参数**：
- `orderId`：订单ID

**响应**：
```json
{
  "success": true,
  "result": [
    {
      "id": "审核记录ID",
      "orderId": "订单ID",
      "stepOrder": 1,
      "roleName": "审核角色名称",
      "auditStatus": "approved",
      "auditUserName": "审核人姓名",
      "auditTime": "2025-01-01 10:00:00",
      "auditRemark": "审核备注"
    }
  ]
}
```

### 3. 工商任务相关接口

参考 `GhBusinessTaskController` 中的接口：
- `/businessTask/managerAudit`：工商经理审核
- `/businessTask/receive`：接收任务
- `/businessTask/updateCost`：更新成本信息
- `/businessTask/handover`：交接任务
- `/businessTask/confirmHandover`：确认交接完成
- 其他接口...

## 配置说明

### 1. 配置审核流程

1. 进入"审核流程管理"页面
2. 创建新的审核流程，例如"订单审核流程"
3. 配置审核步骤：
   - 步骤1：经理审核（角色：经理）
   - 步骤2：出纳审核（角色：出纳）
4. 保存审核流程

### 2. 订单关联审核流程

在创建或编辑订单时：

1. 选择"审核流程"字段
2. 选择已配置的审核流程
3. 保存订单后，系统会自动初始化审核记录

### 3. 灵活配置

审核流程管理支持：

- **多步骤审核**：可以配置多个审核步骤
- **并行审核**：相同步骤顺序的多个角色可以并行审核
- **角色灵活配置**：每个步骤可以关联不同的角色
- **流程启用/禁用**：可以启用或禁用审核流程

## 注意事项

1. **订单创建时自动创建工商任务**：
   - 只有当交单方式是工商部（`deliveryMethod = 'gsb'`）时才会自动创建
   - 如果订单已存在工商任务，不会重复创建

2. **审核流程初始化**：
   - 只有在订单配置了 `auditProcessId` 时才会初始化审核流程
   - 如果审核流程配置不存在或没有步骤，不会初始化

3. **审核状态同步**：
   - 订单审核完成后，如果审核通过且交单方式是工商部，会自动创建或更新工商任务
   - 工商任务的创建不会影响订单审核流程

4. **成本信息同步**：
   - 工商任务中填写的成本信息会自动同步到订单支出表
   - 更新成本信息时，会同时更新订单支出记录

## 扩展说明

如果需要扩展审核流程：

1. **添加新的审核步骤**：在审核流程管理中配置
2. **添加新的审核角色**：在角色管理中创建
3. **自定义审核逻辑**：修改 `GhOrderAuditServiceImpl.executeAudit` 方法
4. **添加审核通知**：在审核完成后发送通知（邮件、短信等）

## 总结

通过订单审核流程与工商任务流程的集成，实现了：

1. **灵活的审核流程配置**：通过审核流程管理可以灵活配置审核步骤和角色
2. **自动化的任务创建**：订单审核通过后自动创建工商任务
3. **完整的流程跟踪**：记录每一步审核和任务处理过程
4. **成本信息自动同步**：工商任务中的成本信息自动同步到订单支出

整个流程既灵活又可配置，能够适应不同的业务需求。
